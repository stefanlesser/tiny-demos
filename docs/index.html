There is a command line utility called gzexe 10 Dec 2025

From the gzexe man page:

The gzexe utility uses gzip(1) to compress executables, producing executables that decompress on-the-fly when executed. This saves disk space, at the cost of slower execution times. The original executables are saved by copying each of them to a file with the same name with a ‘~’ suffix appended. After verifying that the compressed executables work as expected, the backup files can be removed.

I had no idea that there is a tool to compress executables that apparently comes with macOS (and Linux as well as other Unix-based OSes).

Sure, the demoscene has their own kind of specialized tools for compressing executables — I heard of (re)kkrunchy, squishy, and Crinkler so far. But these all seem to be specialized on Windows executables and won’t help me with macOS binaries on arm64 which are MachO.

The more extensive Linux Command Library entry on gzexe mentions:

Its usage has decreased over time due to increasing disk capacities and the emergence of more specialized executable compressors like UPX.

For now gzip-based gzexe seems fine. I should check out UPX as well.

64K demos can call pretty much any OS-provided API 09 Dec 2025

As I was wondering about the rules for 64K demos, I tried to find information about what is officially allowed to be called from the demo executable. Turns out it can call all kinds of APIs, as long as they ship with the operating system.

On Windows that often means .NET/DotNet framework and DirectX graphics APIs that ship with Windows are fair game, as it states here in the Revision 2025 PC Competitions rules:

  • DotNet Entries: DotNet 4.8 that comes with Windows 10 Version 22H2 will be available. We will not install any other versions. […]
  • The compo machine will be installed with DX12.

So calling into dynamically loaded libs (DLLs) is fine. For my macOS experiments I’ll interpret that as calling into any Apple-provided frameworks that ship with macOS is fine.

What ultimately counts is just the size of the executable.

Demos for macOS (or iOS) aren’t really a thing 08 Dec 2025

Let’s get this out of the way: I’m aware that demos for macOS aren’t really a thing. Why, I don’t know, but probably for similar reasons why games for macOS weren’t really a (big) thing either.

The demoscene focuses on all kinds of retro platforms, like C64 or Amiga for instance, which I started my computing “career” on (if you can call it that). Or it happens mainly on Windows, for modern new-school demos, even though the occasional Linux and Mac demo does exist. It appears though that most demos that run on macOS have been ported, or run through cross-platform layers like SDL.

Well, I don’t have a Windows computer and don’t know much about DirectX or any other Windows frameworks for that matter. I do know a bit about various macOS/iOS frameworks, and I’m going to put that knowledge to good use, whether it’s a thing or not. And I’m not trying to make it a thing either. I just want to find something that makes coding fun for me again.
